# Исправление проблемы с превышением лимита PubMed API (HTTP 429)

## Проблема

При параллельной обработке нескольких препаратов одновременно возникали ошибки **HTTP 429 (Too Many Requests)** от PubMed API. Это происходило потому, что:

1. При параллельной обработке 12 препаратов одновременно, каждый делает по 8-10 запросов к PubMed
2. Rate limiter работал правильно, но при большом количестве параллельных потоков общий лимит все равно превышался
3. PubMed рекомендует не более 3 запросов в секунду без API ключа, и с API ключом можно больше, но все равно есть лимиты

## Решение

### 1. Уменьшен лимит запросов

**Было:** 30 запросов в минуту  
**Стало:** 20 запросов в минуту (более консервативный лимит)

```python
# config.py
pubmed_rate_limit_per_minute: int = 20  # Консервативный лимит
```

### 2. Добавлена минимальная задержка между запросами

Даже если rate limiter разрешает запрос, добавляется минимальная задержка (0.1 секунды по умолчанию). Это помогает избежать превышения лимита при параллельной обработке.

```python
# config.py
pubmed_min_delay_seconds: float = 0.1  # Минимальная задержка между запросами
```

### 3. Улучшено логирование

Rate limiter теперь логирует предупреждения (WARNING) вместо debug сообщений при достижении лимита, чтобы было видно, когда система ждет.

### 4. Исправлена логика rate limiter

Убедились, что вызов добавляется в историю ПЕРЕД возвратом из функции, что обеспечивает правильный подсчет.

## Изменения в коде

### `config.py`
- Уменьшен `pubmed_rate_limit_per_minute` с 30 до 20
- Добавлен `pubmed_min_delay_seconds: float = 0.1`
- Добавлена поддержка переменной окружения `PUBMED_MIN_DELAY_SECONDS`

### `modules/api_clients.py`
- Метод `_run_esearch()` теперь использует `wait_if_needed()` напрямую вместо `RateLimiterContext`
- Добавлена минимальная задержка после проверки rate limiter
- Метод `_get_article_details()` также обновлен аналогичным образом
- Добавлено сохранение `_min_delay` в экземпляре клиента

### `modules/rate_limiter.py`
- Улучшено логирование: используется `log.warning` вместо `log.debug` при достижении лимита
- Добавлено более информативное сообщение с упоминанием ошибки 429

## Настройка через переменные окружения

Можно настроить через `.env` файл:

```env
# Лимит запросов в минуту (по умолчанию: 20)
PUBMED_RATE_LIMIT_PER_MINUTE=20

# Включить/выключить rate limiting (по умолчанию: true)
PUBMED_RATE_LIMIT_ENABLED=true

# Минимальная задержка между запросами в секундах (по умолчанию: 0.1)
PUBMED_MIN_DELAY_SECONDS=0.1
```

## Рекомендации PubMed API

Согласно официальным рекомендациям NCBI E-utilities:

1. **Без API ключа:** не более 3 запросов в секунду (180/мин)
2. **С API ключом:** можно больше, но рекомендуется консервативный подход
3. **При параллельной обработке:** необходимо дополнительное ограничение

Наша реализация:
- Использует консервативный лимит 20 запросов/мин
- Добавляет минимальную задержку 0.1 секунды между запросами
- Правильно обрабатывает параллельные запросы через thread-safe rate limiter

## Проверка работы

После исправлений система должна:
1. ✅ Автоматически ждать при достижении лимита
2. ✅ Добавлять минимальную задержку между запросами
3. ✅ Логировать предупреждения при блокировке запросов
4. ✅ Избегать ошибок HTTP 429

## Мониторинг

В логах теперь будут видны предупреждения:
```
PubMedRateLimiter: Rate limit reached (20/20). Waiting X.XXs to avoid 429 error...
```

Это нормальное поведение - система автоматически ждет, чтобы не превысить лимит.

## Дополнительные рекомендации

Если ошибки 429 все еще возникают:

1. **Уменьшите количество параллельных воркеров:**
   ```bash
   python main.py -i "protocol.docx" -o "report.xlsx" --indication "Disease" --max-workers 4
   ```

2. **Увеличьте минимальную задержку:**
   ```env
   PUBMED_MIN_DELAY_SECONDS=0.2
   ```

3. **Уменьшите лимит запросов:**
   ```env
   PUBMED_RATE_LIMIT_PER_MINUTE=15
   ```
